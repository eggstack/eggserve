# Plan 082 — HEAD, Error-Response, and Validator Correctness

## Goal

Make HEAD behavior consistent with the corresponding GET across successful file responses, ranges, directory listings, redirects, and errors; ensure body-forbidden statuses never transmit payload bytes; and strengthen validators so rapid same-size modifications do not silently reuse stale entity tags where platform metadata can distinguish the file state.

## Preconditions

- Plan 081 has delivered one canonical static-file representation planner and body-construction path.
- Plans 077–080 have stabilized timeout, body, service, and configuration contracts.
- Existing canonical response normalization and raw-wire test harnesses are available.

## Non-goals

Do not:

- add new HTTP methods;
- implement multipart byte ranges unless already supported;
- add cache storage or proxy behavior;
- redesign directory listing presentation;
- claim strong ETags unless byte-for-byte identity semantics are proven;
- complete Windows directory-handle traversal assigned to Release D;
- optimize file streaming.

## Defect statement

HEAD behavior is currently correct in portions of the successful file path but inconsistent in error helpers and directory-listing handling. Some paths can calculate `Content-Length` from the empty HEAD body rather than the corresponding GET representation, while generic error helpers may construct payload bodies without receiving method context.

Existing weak validators rely primarily on size and coarse modification time. Same-size rewrites within one timestamp second can therefore collide, causing stale conditional responses.

## Track A — Define the canonical HEAD contract

Document the project-wide contract:

- evaluate a HEAD request using the same resource resolution, preconditions, range rules, status selection, and representation metadata as GET;
- transmit no message body;
- preserve headers describing the corresponding GET representation where the protocol permits;
- set `Content-Length` to the GET payload length when known and allowed, not zero merely because HEAD sends no body;
- preserve 206/304/416 semantics according to the project's canonical HTTP policy;
- never create or poll a response body stream solely for HEAD;
- apply the same access-control and error decisions as GET.

Clarify behavior for body-forbidden statuses such as 1xx, 204, and 304 separately from HEAD.

## Track B — Central response normalization

Create or strengthen one final normalization stage that receives:

- request method;
- selected status;
- planned representation length;
- planned response headers;
- optional body source;
- protocol/version context where required.

The normalizer must:

- suppress body creation/transmission for HEAD;
- suppress bodies for body-forbidden statuses;
- preserve or remove `Content-Length` according to the canonical policy for each status;
- avoid conflicting `Transfer-Encoding` and `Content-Length` output;
- avoid acquiring file-stream permits for suppressed bodies;
- leave connection framing unambiguous.

Do not rely solely on Hyper or the client library to discard a body after application code has created it. The application response object should already reflect the no-body plan.

## Track C — Error response unification

Inventory every error response constructor and call site:

- bad request;
- forbidden;
- not found;
- method not allowed;
- request body rejected/too large;
- range unsatisfiable;
- internal server error;
- service unavailable/resource saturation;
- redirects if present;
- TLS or parser errors that produce HTTP responses.

Replace helpers that cannot receive method/normalization context with a typed error response plan or route all errors through the final normalizer.

For HEAD errors:

- status and headers should match GET;
- no payload bytes may be transmitted;
- `Content-Length` should represent the corresponding generated GET error body where the project chooses to expose one;
- sensitive internal diagnostics must not enter either GET or HEAD payloads.

## Track D — Directory listing HEAD semantics

Separate directory listing representation generation from body transmission.

Preferred behavior:

- calculate the exact GET listing bytes or exact length using the same bounded generator;
- for HEAD, expose GET headers and representation length without constructing a network body;
- preserve listing entry/byte limits;
- avoid expensive duplicate traversal merely to calculate a length if a bounded single generation can be retained or measured safely;
- release listing permits promptly.

If exact length cannot be known without generating the listing, generation is acceptable within the same configured limits, but the generated bytes must not be streamed for HEAD.

Test empty, small, maximum-boundary, denied, and over-limit listings.

## Track E — Range and conditional HEAD behavior

Using the Plan 081 planner, verify:

- HEAD with a satisfiable range follows the project's selected 206 policy and reports the range representation length without a body;
- unsatisfiable range produces the same 416 headers as GET without a body;
- matching validators produce 304 with no body;
- `If-Range` match/mismatch follows GET status/header selection;
- conditional precedence does not change because the method is HEAD;
- no file read occurs for HEAD after metadata/planning is complete.

Add instrumentation to prove no stream body or file read loop is created.

## Track F — Validator requirements

Define weak ETag inputs by platform.

The validator should incorporate, where available and stable:

- file length;
- highest available modification-time precision;
- Unix device and inode identity;
- Windows volume serial/file identifier;
- additional change-time or generation metadata only if reliable and documented.

Requirements:

- derive metadata from the opened object used by Plan 081;
- encode deterministically and safely;
- remain weak unless byte identity across metadata changes is guaranteed;
- avoid leaking sensitive absolute paths;
- produce valid quoted ETag syntax;
- behave consistently across direct and index URL forms;
- document filesystem/platform limitations.

Do not hash complete file contents on every request. Content hashing would be a separate cache/performance design.

## Track G — Timestamp handling

Audit Last-Modified conversion and comparison:

- use available precision internally;
- serialize HTTP dates at required protocol precision;
- apply comparison semantics consistently despite HTTP-date second granularity;
- prefer ETag precedence where both validators are supplied;
- handle pre-epoch, out-of-range, or unavailable timestamps safely;
- ensure rapid updates do not rely solely on Last-Modified when ETag metadata can distinguish them.

## Track H — Same-object consistency

Ensure response validators, length, and body all describe the same opened object.

Tests should cover:

- pathname replacement after resolution;
- same-size rewrite through another handle;
- truncate/extend while a handle is open;
- direct/index URL comparison;
- platform differences in metadata update behavior.

Where a filesystem cannot provide change-distinguishing metadata, document the validator as conservative and avoid unsupported guarantees.

## Track I — Body-forbidden status matrix

Create a reusable matrix for:

- HEAD with ordinary statuses;
- 1xx where server-generated responses exist;
- 204;
- 304;
- successful CONNECT only if applicable, otherwise explicitly out of scope;
- error statuses.

Assert:

- payload bytes sent;
- `Content-Length` policy;
- `Transfer-Encoding` absence/presence;
- keep-alive framing;
- body factory invocation count;
- file/listing permit acquisition count.

## Required tests

### Canonical unit tests

For each representative status, compare GET and HEAD plans:

- status equality;
- representation headers;
- content length;
- body-required flag;
- range/conditional outcome;
- no stream allocation for HEAD.

### Raw-wire tests

- HEAD regular file;
- HEAD range;
- HEAD 304;
- HEAD 416;
- HEAD directory listing;
- HEAD redirect if supported;
- HEAD 400/403/404/405/413/500/503;
- pipelined or keep-alive request following each HEAD response;
- body-forbidden statuses followed by another request to detect framing errors;
- HTTP/1.0 and HTTP/1.1 where supported.

### Validator tests

- same size and same whole-second mtime but distinguishable high-resolution metadata;
- rapid replacement with new inode/file identity;
- direct and index URL share validator;
- unchanged opened file retains validator;
- invalid/unavailable metadata fallback;
- ETag syntax and conditional matching;
- platform-specific Unix and Windows identity evidence.

## Documentation changes

Update:

- HTTP semantics/conformance reference;
- HEAD and range behavior;
- directory listing behavior;
- validator/ETag design note;
- platform/filesystem limitations;
- release criteria and test matrices;
- finding registry and corrective status.

## Acceptance criteria

- Every response path is normalized with request-method context.
- HEAD transmits zero payload bytes for success, range, listing, redirect, and error paths.
- HEAD preserves the corresponding GET status and representation headers according to documented policy.
- HEAD `Content-Length` is not incorrectly forced to zero when the GET representation length is known.
- No file-stream or listing body is created solely for HEAD.
- Body-forbidden statuses cannot carry application payload bytes or ambiguous framing.
- Direct and directory-index forms share validators.
- ETags use high-resolution and platform identity metadata where available and do not rely only on size plus whole-second mtime.
- Rapid same-size replacement tests change the validator where the platform exposes distinguishing metadata.
- Raw-wire keep-alive tests prove correct framing after HEAD and body-forbidden responses.
- Exact-SHA platform evidence and installed-artifact evidence are recorded.

## Stop conditions

Stop and add a blocking finding if:

- an outer transport layer reintroduces payload bytes after application normalization;
- error responses cannot be routed through method-aware normalization;
- directory listing length calculation requires bypassing configured work limits;
- validator metadata is obtained from a different object than the streamed body;
- a supported platform cannot provide any stable validator input beyond collision-prone coarse metadata without documentation/support narrowing.

## Handoff

Plan 083 performs independent canonical, raw-wire, production-path, and artifact-level closure for Release C and reruns the body/lifecycle gates invalidated by response normalization changes.
