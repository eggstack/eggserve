# Plan 027: Rust-Owned Python Server Primitives

## Purpose

Expose Python-facing server primitives that allow Python code to build small dynamic/static HTTP servers while Rust owns the socket accept loop, HTTP parsing, response serialization, file streaming, backpressure, concurrency limits, and timeout enforcement.

This plan is the key downstream-enablement step for future external ASGI/WSGI/app-server projects. The eggserve repo must not implement ASGI or WSGI. Instead, it should expose low-level Python primitives that an external adapter can build on.

## Current context

The Python package currently provides:

- Native path/static primitives when the PyO3 extension is available.
- A subprocess API for running the Rust binary.
- `ServeConfig`, `StaticPolicy`, `ServerProcess`, and `serve_directory`.

It does not yet expose a Rust-owned in-process server loop with Python request dispatch. That is intentional so far. This plan adds the minimal primitive needed for downstream dynamic server construction without turning eggserve into a framework.

## Goals

- Add a Rust-owned server lifecycle exposed to Python.
- Allow Python callbacks/handlers to choose explicit response objects.
- Keep socket I/O, HTTP parsing, concurrency, timeouts, response serialization, and static file streaming in Rust.
- Support static file responses through safe body sources from Plan 026.
- Support byte/text/empty responses for simple dynamic endpoints.
- Provide enough structure for downstream ASGI/WSGI adapters to be built externally.
- Maintain safe defaults and explicit weakening rules.

## Non-goals

- Do not implement ASGI.
- Do not implement WSGI.
- Do not implement routing beyond optional examples or trivial user code.
- Do not implement middleware.
- Do not implement template rendering.
- Do not implement request body streaming into Python in the first pass.
- Do not implement WebSocket.
- Do not implement HTTP/2.
- Do not implement long-lived background task management.
- Do not sandbox Python code.

## Design model

The design should be primitive and explicit.

Rust provides:

- Listener binding.
- Connection accept loop.
- Connection semaphore.
- Header read timeout.
- Write timeout.
- Graceful shutdown.
- HTTP request parsing through Hyper.
- Request body metadata policy.
- Response serialization.
- Static file/range streaming.
- Error handling for malformed protocol input.

Python provides:

- A handler callable.
- Policy decisions for dynamic responses.
- Optional use of `SecureRoot`/`StaticResponder` to produce static responses.
- Explicit response objects: empty, bytes, text, static body source.

The Python callback receives request metadata, not a raw socket.

## Candidate Python API

This is illustrative, not mandatory:

```python
from eggserve import Server, ServerConfig, Response, StaticResponder

static = StaticResponder('public')

def handle(request):
    if request.path == '/health':
        return Response.text('ok', content_type='text/plain')
    return static.respond(request)

server = Server(ServerConfig(bind='127.0.0.1', port=8000), handle)
server.serve_forever()
```

Alternative names are fine if consistent with existing `ServeConfig`. Avoid names that imply ASGI/WSGI compatibility.

## Request object requirements

The Python request object should be immutable and expose:

- `method`.
- `target` or `raw_target`.
- `path`.
- `query` if parsed cheaply and safely.
- `headers` as an immutable mapping or list preserving enough information for downstream use.
- `remote_addr` if available.
- `http_version` if useful.
- `has_body` or body metadata status.

For this first pass, do not expose a streaming request body. Static server rules may reject bodies by default. If allowing dynamic handlers to accept bodies is desired later, create a separate plan for bounded body aggregation/streaming.

## Response object requirements

Python response objects should be explicit and immutable after construction where practical:

- `Response.empty(status=204, headers=None)`.
- `Response.bytes(data, status=200, headers=None, content_type=None)`.
- `Response.text(text, status=200, headers=None, content_type='text/plain; charset=utf-8')`.
- `Response.body_source(body, status, headers)` or equivalent for static file/range bodies.
- `Response.from_plan(plan, body_source)` for static responses.

Response validation must happen before Rust writes to the socket:

- Status code must be valid.
- Header names must be valid.
- Header values must not contain invalid control characters.
- Hop-by-hop or unsafe headers should be handled according to documented policy.
- Content-Length must match body source or be generated by Rust.
- Python should not be allowed to lie about content length for Rust-owned body sources.

## Static responder primitive

Add a Python helper around `SecureRoot` and response planning. It should not be a router or framework. It should be a convenience wrapper for the common safe static case.

Candidate API:

```python
static = StaticResponder('public', policy=StaticPolicy())
response = static.respond(request)
```

Behavior:

- Uses the same path parsing and filesystem policy as the CLI.
- Serves files and ranges using body sources.
- Serves index files if current CLI behavior does.
- Applies directory listing policy if enabled.
- Returns explicit 403/404/405/416 response objects.
- Does not expose absolute filesystem paths.

If implementing full static responder parity is too large for the first pass, start with file-only static responses and document directory behavior as deferred.

## Rust implementation tasks

### 1. Add a server module or crate boundary

Decide whether this belongs in `eggserve-core` or a separate crate. Prefer keeping the first pass in existing crates unless the boundary becomes muddy.

The core primitive should be framework-neutral:

- A request metadata type.
- A response value type.
- A callback/handler trait or internal bridge.
- Server config with safe defaults.
- Runtime lifecycle methods.

### 2. PyO3 callback bridge

Implement a PyO3 bridge that calls a Python handler under the GIL and converts the returned Python `Response` into a Rust response representation.

Important concerns:

- Handler exceptions should map to 500 responses by default without leaking stack traces to clients.
- Exceptions should be logged/surfaced to the caller where possible.
- Python callbacks may block. Document this and consider using `spawn_blocking` or a bounded callback executor if needed.
- Do not allow unbounded concurrent Python callbacks by default. Add a `max_python_callbacks` limit or reuse connection limits conservatively.
- Avoid holding the GIL during Rust file streaming.

### 3. Server lifecycle

Expose:

- Blocking `serve_forever()`.
- `start()` / `stop()` lifecycle object if feasible.
- Context-manager support if simple.
- `addr` or `bound_addr` for port 0 tests.

The implementation should support test usage with `127.0.0.1:0`.

### 4. Configuration

Use safe defaults matching the CLI:

- Bind to `127.0.0.1`.
- Require explicit `public=True` for unspecified bind addresses.
- Directory listing disabled.
- Symlink following denied.
- Dotfiles denied.
- Conservative max connections.
- Conservative max file streams.
- Header timeout.
- Write timeout.
- Optional TLS later or only if easy to route through existing TLS feature.

Do not silently differ from CLI defaults.

### 5. Response serialization

Rust must serialize responses. Python response objects should not contain raw HTTP bytes. Rust should generate or validate:

- `Content-Length`.
- `Content-Type`.
- `Connection` behavior where needed.
- Header validity.
- Empty-body rules for HEAD.
- Static range headers.

## Python tests

Add tests for:

- Server starts on loopback and responds to a dynamic text endpoint.
- Server returns static file through `StaticResponder` or body source.
- HEAD returns no body with correct headers.
- Python handler exception produces 500 without leaking traceback in body.
- Invalid response header is rejected or maps to 500 safely.
- Public bind without `public=True` fails at config construction.
- Server can bind to port 0 and expose actual bound address.
- `stop()` terminates the server.
- Concurrent requests respect configured limits where practical.

## Rust tests

Add tests for:

- Handler bridge maps request metadata correctly.
- Response conversion rejects invalid status/header values.
- Body source response conversion preserves content length.
- Python callback errors do not panic Rust.
- GIL is not held during static file streaming if testable or reviewable.

## Acceptance criteria

- Python can start an in-process Rust-owned server with a Python handler.
- Rust owns socket I/O, HTTP parsing, response serialization, concurrency, and timeouts.
- Python can return text/bytes/empty responses.
- Python can return static file/range responses through safe body sources without path reopening.
- Handler exceptions fail closed as 500 without leaking local paths or Python traceback to the HTTP client.
- Public bind requires explicit acknowledgement.
- No ASGI/WSGI/router/middleware/framework code is added.
- Docs explain how downstream adapters can build externally on these primitives.
- Full Rust and Python validation passes.

## Handoff notes

This plan is intentionally close to an application server boundary but must not cross it. The primitive is: Python handler returns explicit response values while Rust owns I/O. Anything involving ASGI scopes, WSGI environ, middleware chains, routing, or framework lifecycle belongs in a downstream project.
